# AI Instructions - Trisilar Task Hub

**Doc Role:** Universal operating rules for every AI agent in this repo
**Status:** Active
**Owner:** PM
**Last Updated:** 2026-05-15
**Updated by:** Codex PM / Integration Owner

This file is the first document every AI agent must read before doing PM, Dev, QA, or integration work in this repository. Agent-specific files such as `CODEX.md`, `CLAUDE.md`, and `GEMINI.md` extend these rules. They do not replace them.

---

## Read Order

Read only what is needed for the current task, but start in this order:

1. `.ai-instructions.md`
2. `docs/reference/PROJECT_CONTEXT.md`
3. `CURRENT_SPRINT.md`
4. `docs/plans/VERSION_0_2_PLAN.md` or the active version plan
5. `docs/plans/VERSION_0_2_PARALLEL_WORKSTREAM_PROMPTS.md` when running W1/W2/W3
6. Agent-specific file: `CODEX.md`, `CLAUDE.md`, or `GEMINI.md`
7. `docs/reference/AI_AGENT_GOVERNANCE.md` and `docs/agents/` when the task is role-specific V0.3 work
8. `docs/reference/BRANCH_ENVIRONMENT_WORKFLOW.md`
9. `docs/reference/FILE_ORGANIZATION.md` when adding, moving, or renaming files
10. Task-specific architecture, ADR, testing, design, or deployment docs

For large code files, grep/search first and read targeted ranges only.

## Project Planning Terms

| Term | Meaning | Primary file |
|---|---|---|
| Version | A releasable product milestone, such as V0.1 or V0.2 | `docs/plans/VERSION_0_2_PLAN.md` |
| Phase | A sequential capability block inside a version | version plan or `docs/archive/DEVELOPMENT_HISTORY.md` for history |
| Workstream | A parallel track inside a version, such as W1/W2/W3 | version plan and prompt registry |
| Task | A concrete implementation or review unit | `CURRENT_SPRINT.md` next action |
| QA Round | Evidence-backed verification for a task | `docs/logs/QA_LOG.md` |
| ADR | A durable architecture/process decision | `docs/adr/` |

Rules:

- `CURRENT_SPRINT.md` is not the roadmap. It is the short current-state and next-action file.
- `TODO.md` is the high-level roadmap index. It must point to active plans, not duplicate all details.
- Version plans own scope, phases, workstreams, dependencies, and release logic.
- ADRs own decisions that affect architecture, process, data contracts, deployment, or multi-agent workflow.
- QA evidence belongs in `docs/logs/QA_LOG.md`, not in `CURRENT_SPRINT.md`.

## Task ID Rules

Use one canonical task ID format for V0.2 work:

```text
V0.2-W<workstream>-<two-digit sequence>
```

Examples:

- `V0.2-W1-05` for W1 company access / deployment work.
- `V0.2-W2-02` for W2 Review Queue redesign.
- `V0.2-W3-01` for W3 Paperclip contract / mock adapter work.

Rules:

- Workstream labels such as `W1`, `W2`, and `W3` remain valid for high-level planning.
- Old short labels such as `W1.4`, `W2b`, and `W3 sequence 1` are aliases only. Put the canonical ID first.
- Do not create new active IDs such as `W3-P0`, `P7`, or `P8` for V0.2 work. Use `Priority: P0/P1/P2` separately when priority is needed.
- Historical docs and QA logs may keep their original labels. Do not rewrite history only to rename old tasks.
- New prompts, commits, QA reports, PM updates, and PR notes must lead with the canonical ID, then include the alias when useful.

For V0.3 Product Reliability + UX Stabilization work, use:

```text
V0.3-RUX-<two-digit sequence>
```

Examples:

- `V0.3-RUX-03` for Review Queue and AI trace clarity.
- `V0.3-RUX-06` for the `dev -> main` release checklist.

After a version is complete, keep these IDs as historical references. Do not reopen completed V0.2/V0.3 work without a new PM route.

## Role Rules

Each session has one role.

| Role | May do | Must not do |
|---|---|---|
| Dev | Implement, verify, commit, push | Perform final QA signoff or rewrite PM trackers |
| Dev Fix | Fix a QA-confirmed issue, verify, commit, push | Expand scope beyond the defect |
| QA | Inspect, run checks, report PASS/FAIL with evidence | Edit files |
| QA Recheck | Recheck a specific fix with evidence | Edit files |
| PM | Update planning docs, decide next task, maintain conventions | Implement product behavior or perform QA signoff |

If the user does not specify a role and the request is operational, infer the smallest safe role. If the request could cause a process conflict, ask for clarification.

For V0.3 long-term role ownership, use `docs/reference/AI_AGENT_GOVERNANCE.md` and the relevant file under `docs/agents/`. The table above remains the minimal legacy role model for general Dev/QA/PM sessions.

## Branch and Worktree Rules

Use `docs/reference/BRANCH_ENVIRONMENT_WORKFLOW.md` as the source of truth.
Use `docs/reference/CODEX_PARALLEL_DEVELOPMENT_MODEL.md` for V0.3 role-based parallel Codex work.

Required branch flow:

```text
topic branch -> dev -> main
```

Allowed topic branch namespaces:

| Prefix | Use |
|---|---|
| `feature/*` | PM-assigned product/workstream feature branches and historical W1/W2/W3 branches |
| `codex/*` | Codex-created implementation, docs, QA, integration, backup, or release-prep branches |
| `claude/*` | Claude-created implementation/docs branches when Claude is acting as the agent |
| `hotfix/*` | Emergency production fixes from `main`, then backport to `dev` |

Parallel work must use separate git worktrees:

```powershell
git worktree add ..\trisilar-task-hub-w1-company-access feature/w1-company-access-deployment
git worktree add ..\trisilar-task-hub-w2-ui-redesign feature/w2-02-review-redesign
git worktree add ..\trisilar-task-hub-w3-paperclip feature/w3-paperclip-integration
git worktree add ..\trisilar-task-hub-v03-release-qa -b codex/v03-branch-workflow-release-qa origin/dev
```

Before editing, run:

```powershell
git status --short --branch
```

Rules:

- Do not run multiple active workstreams in the same working directory.
- Do not commit W1/W2/W3 or V0.3 feature work directly on `main`.
- Do not share one feature branch across different workstreams.
- Do not reuse a completed branch/worktree for a new phase; create a new scoped branch or explicitly reattach to an existing PM-assigned branch.
- Backup branches such as `codex/backup-*` are safety refs only. Do not use them as PR sources unless PM explicitly asks.
- Stage specific files only. Never use `git add .`.
- Never revert unrelated changes from another user or agent.
- If the worktree is dirty with unrelated files, leave them alone and stage only your owned files.

## Encoding Rules

- Markdown files may contain Thai, emoji, or typographic punctuation.
- Use targeted UTF-8-safe edits.
- Avoid broad PowerShell default-encoding rewrites.
- Avoid copy/paste transformations that can create mojibake.
- If mojibake already exists, do not rewrite the file unless the task is specifically to repair it.

## Verification Rules

For code, config, route, integration, or behavior changes:

```powershell
node server.js
npm.cmd run check:all
```

Run `node server.js` in a separate terminal before `check:all`, because smoke tests call local HTTP endpoints.

For docs-only changes:

```powershell
git diff --check
```

Docs-only work may skip runtime checks when no behavior/config files changed. State that explicitly in the final handoff.

Every meaningful Dev/QA/PM update in project docs must identify the responsible agent.

```text
Implemented by: <agent name>

Updated by: <agent name>
Commit: <hash>
Verification: <commands and result>
```

```text

Branch:
Worktree:
Owned files:

```

## Stop Conditions

Stop and ask PM before continuing when:

- The task requires editing another active workstream's files.
- The task requires changing branch strategy, environment model, data contract, auth/access policy, or release gate.
- Verification contradicts the expected state.
- A secret, token, credential, or production-only dependency is required.
- The repo is dirty in a way that prevents a clean, scoped commit.
